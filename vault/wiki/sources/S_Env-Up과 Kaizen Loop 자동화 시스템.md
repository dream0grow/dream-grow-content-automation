# Env-Up과 Kaizen Loop — Claude Code 자동화 시스템

Claude Code 환경 자동 점검 + 에이전트 품질 자동 측정 두 시스템에 대한 운영 메모. LLM 자가평가의 한계를 외부 스크립트로 보완하는 [[C_측정과 판단의 분리]] 원리가 핵심.

## 첫 번째: Env-Up — 외부 변화 추적

Claude Code 신규 버전이 내 시스템에 미치는 영향을 자동 점검.

1. `claude --version`으로 현재 버전 확인
2. GitHub 릴리즈 노트 자동 수집
3. 추적 중인 Known Issues 46건 전수 대조
4. "이거 해결됐네?" → 우회 코드 제거 대상 표시
5. "이 신기능으로 내 Hook 대체 가능?" → 기능 교체 검토

**효과**: v2.1.97~101 네 개 버전 한 번에 처리, 이슈 4건 해결 + 보안 강화 3건 감지. 수동 반나절 → 커맨드 한 번.

## 두 번째: Kaizen Loop — 내부 품질 측정

파이썬 스크립트 실행으로 9축 자동 측정.

### AQI 6축 (에이전트/스킬 품질)
- 위임 정확도: 적합한 에이전트에게 맡겼나
- 역할 집중도: 딴짓 안 했나
- 제약 준수율: 규칙 잘 지켰나
- 산출물 품질: 결과물 쓸만한가
- 컨텍스트 효율: 토큰 낭비 없었나
- 비용 효율: 돈값 했나

### 인프라 3축 (운영 건강도)
- Hook 건강도 (30개 Hook 에러 없이 도는가)
- 규칙 파일 건강도 (문서 최신, 모순 없음)
- 운영 병목 (MCP 속도, 압축 과다)

**측정 사례**: AQI 81.4점, Hook이 에러를 몰래 삼키고 있던 72건 적발 → 즉시 수정 명령.

## 핵심 원리 — [[C_측정과 판단의 분리]]

> "LLM한테 '너 스스로 평가해봐'라고 시키면 자기가 만든 걸 자기가 채점하는 거라 항상 후하게 나옵니다."

스크립트가 냉정하게 숫자를 매기고, LLM은 그 숫자를 해석하고 고치는 데만 사용. 측정과 판단을 분리하면 사용자는 의사결정권자 위치에서 승인만 하면 된다.

## Dream_Grow 시스템 적용 가능성

- 콘텐츠 자동화 파이프라인의 산출물 품질을 외부 스크립트로 점수화
- diff_learner.py와 결합하여 수정 빈도/수정 범위로 AQI 컨텍스트 효율 측정
- launchd 정기 작업의 Hook 건강도 모니터링
