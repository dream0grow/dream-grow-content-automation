# 노션 → 옵시디언 전면 이관 설계

> 작성: 2026-07-06 · 결정: **전부 옵시디언 중심으로 재구축, 노션 철수** (비용 절감 + SSOT 단일화)
> 전제: 볼트 = 이 저장소 `vault/` (GitHub 동기화, Obsidian Git). 플라우드 파이프라인은 이미 볼트 기반.
> 이 문서는 다음 세션의 구현 사양서다. 여기 없는 결정은 사용자에게 묻는다.

## 0. 목표와 원칙

1. **볼트가 유일한 진실원천**: 카드(콘텐츠 단위)의 상태·본문·이력 전부 `vault/` 안 md 파일.
2. **발행 로직 재사용**: Threads 체인·스티비 발송 코드는 그대로. 읽는 곳만 노션→볼트로 스왑.
3. **상태 머신 보존**: 검증된 stage 흐름(intake→research→keyword→…→publish)을 바꾸지 않는다.
   저장소만 바꾼다. 승인 게이트 2곳(키워드·발행)도 동일.
4. **끝나면 노션 구독 해지**: 병행 운행 기간을 짧게 잡고, 남은 카드를 소진·아카이브한 뒤 해지.

## 1. 현행 노션 의존 지도 (실사)

| 모듈 | 노션 사용 | 치환 대상 |
|---|---|---|
| `notion_state.py` | 카드 CRUD(`query_cards`/`update_card`/`create_card`), 본문 토글(`append_section`/`read_sections`), ID 채번(`next_content_id`), 멘션 알림(`notify`) | **`obsidian_state.py` 신규** — 동일 함수 시그니처 |
| `run.py` DISPATCH | stage/status/approval 조합으로 카드 라우팅 | 변경 없음 (state 모듈만 교체) |
| `daily_intake.py` | 새 주제 카드 생성 | create_card 호출처만 교체 |
| `publish.py` | 카드에서 초안 읽기, 발행 결과 기록 | read/update 호출처만 교체 |
| `rubric_review.py`, `agent_dialogue.py` 등 | 본문 토글 읽기/쓰기 | append/read 호출처만 교체 |
| `.claude/skills/plaud-zettel` | 노션 제텔카스텐 DB | **폐기** — `vault_pipeline`이 대체 완료 |

## 2. 카드 = 볼트 md 파일

위치: `vault/파이프라인/{활성|발행완료|중단}/DG-YYYY-NNNN 제목.md`

```yaml
---
content_id: DG-2026-0021
topic: 아이가 숙제를 미룰 때
stage: keyword_approval        # intake|research|keyword|keyword_approval|brief|draft|approval|publish_ready|published
status: waiting                # queued|running|waiting|needs_human|done|error
format: thread                 # thread|newsletter
audience: 초등 저학년 학부모
approval_status: ""            # approved 입력 = 승인 (사람만)
review_status: ""              # 검수 에이전트가 기록
approved_keyword: ""
idempotency_key: ""
last_error: ""
published_url: ""
updated: 2026-07-06T22:08+09:00
---

## 리서치 (2026-07-06 22:10)
…

## 키워드 표 (2026-07-06 22:20)
…

## ✍️ 초안 (thread) (2026-07-07 04:05)
…
```

- 노션의 "본문 토글" = **md의 `## 제목 (타임스탬프)` 섹션**. `append_section`은 파일 끝에
  섹션 추가, `read_latest_section`은 같은 접두사의 마지막 섹션을 읽는다 — 현행 함수 의미 그대로.
- 노션 select 속성 = frontmatter 문자열 필드. `query_cards(stage=…, status=…)`는
  `파이프라인/활성/*.md`의 frontmatter 스캔(수십 장 규모라 성능 문제 없음).
- `next_content_id()`: 파일명·장부에서 최대 번호+1. 동시 실행은 워크플로우 `concurrency`가 막는다.
- published 카드는 `발행완료/`로 이동(활성 폴더를 얇게 유지).

## 3. `obsidian_state.py` — notion_state와 같은 얼굴

```python
# 시그니처 동일 유지가 이관 비용을 최소화한다
query_cards(stage=None, status=None, approval_status=None) -> list[dict]
update_card(card_path, **fields)          # page_id 대신 파일 경로
create_card(topic, *, stage, status, format, audience, ...) -> path
append_section(card_path, heading, body)
read_sections(card_path) / read_latest_section(card_path, heading_prefix)
next_content_id() -> str
notify(card_path, message)                # ↓ 4장 승인 UX
```

호출처의 `page_id` 변수는 카드 파일 경로 문자열로 그대로 흘려보낸다(타입만 바뀜).
`card_from_page` 상당은 `parse_frontmatter`(vault_pipeline/vault_io.py 재사용).

## 4. 승인 UX — 노션 모바일 앱의 대체

노션에서 하던 두 승인(키워드·발행)을 옵시디언에서 한다:

| 단계 | 방식 |
|---|---|
| **지금(M1)** | 옵시디언(모바일 앱 + Obsidian Git 플러그인, 또는 GitHub 모바일 앱에서 파일 편집)에서 frontmatter `approval_status: approved` 입력 → 다음 cron이 감지해 진행. 승인 대기 목록은 파이프라인이 `_system/review_queue.md`에 집계 |
| **알림** | 노션 멘션 대신: 승인 대기 발생 시 GitHub Actions가 **이슈 생성/갱신**("발행 승인 대기 N건") → GitHub 모바일 앱 푸시 알림. 구현 1시간, 추가 토큰 불필요 |
| **중기(Phase B)** | 통합기획 v2.1 D-23: 텔레그램 봇(버튼 결재 + 음성 지시). 이 설계와 충돌 없음 — 봇은 frontmatter만 바꾸는 리모컨 |

키워드 승인은 현재도 자동(`DG_AUTO_APPROVE_KEYWORD=true`)이므로 실제 사람 게이트는 발행 승인 1곳뿐이다.

## 5. 이관 로드맵

| Phase | 내용 | 분량 |
|---|---|---|
| **M0 준비** | `vault/파이프라인/` 골격 + `obsidian_state.py` + 단위테스트(카드 CRUD·섹션·채번) | 0.5일 |
| **M1 전환** | `run.py`·`daily_intake.py`·`publish.py`·`rubric_review.py`의 import를 교체(환경변수 `DG_STATE_BACKEND=obsidian`으로 스위치 가능하게), orchestrator.yml이 볼트 커밋·푸시하도록 수정, GitHub 이슈 알림 추가 | 1일 |
| **M2 병행 소진** | 신규 카드는 전부 볼트에서. 노션의 미완료 카드(승인 대기 7장 등)는 노션 모드로 마저 발행하거나 볼트로 손 이관. 2주 관찰 | 대기 |
| **M3 아카이브·해지** | 노션 발행완료 카드 전체를 md로 내보내 `vault/파이프라인/발행완료/`에 보존(스크립트) → 노션 API 키 Secret 삭제 → **노션 구독 해지** | 0.5일 |

리스크와 방어:
- **Actions 동시 커밋 충돌**: orchestrator·plaud-pipeline이 같은 저장소에 push →
  두 워크플로우 모두 `git pull --rebase` 후 push + 서로 다른 시각 슬롯(이미 적용).
- **모바일 승인 체감 저하**: 노션 앱보다 한 단계 번거로움 → 4장 GitHub 이슈 알림으로 보완,
  근본 해결은 Phase B 텔레그램.
- **로컬 편집과 자동 커밋 충돌**: Obsidian Git이 pull 우선이면 대부분 자동 병합.
  카드 파일은 자동화가 "섹션 append + frontmatter 갱신"만 하므로 충돌 면적이 작다.
