## PART 0. 사고 과정 기록 (이 결론에 도달한 경로)

병렬 검토 AI는 이 장을 먼저 읽고, 각 단계의 추론에 오류·누락이 없는지 검증할 것.

### 0-1. 맥락 수집

두 개의 독립 프로젝트가 존재한다.

| 항목 | 드림그로우 콘텐츠 자동화 OS | 제텔카스텐 에이전트 OS (초생산) |
|------|---------------------------|-------------------------------|
| 목적 | 스레드 1편 → 뉴스레터·카드뉴스·릴스·유튜브 자동 생성·발행 | 책·논문 집필을 위한 세컨드 브레인 |
| 볼트 | SNS-시스템 폴더 구조 (Obsidian Sync) | 초생산 볼트, 5단계 파이프라인 (메모→K→O→P→두번째뇌) |
| 기존 자산 | diff_learner.py, Honcho 문체학습, scheduled_publisher.py, zettel_reader.py, 프론트매터 상태기계(리뷰대기→리뷰완료→발행완료) | 4,262개 md, Roam 원본(진짜 지식), 8개 에이전트 설계, 사례은행, `_ai` 마킹 규칙 |
| 승인 채널 | 텔레그램 봇 확정 (D-02), 음성 승인 루프 (Phase B) | 아침 루틴 10분 (옵시디언 기반으로 설계됨) |
| 핵심 원칙 | 명시적 확정 원칙, 회수 유예, 원본 연결 의무(D-10) | AI는 의견·주장·사례를 창작하지 않는다. 출처 사슬 의무. `_ai` 메모는 4~5단계 인용 금지 |

### 0-2. 연동 요구와 위험 진단

사용자 요구: "콘텐츠를 만들수록 세컨드 브레인이 성장하는 플라이휠." 발행 콘텐츠에 녹아든 본인의 추가 인사이트를 메모로 환류하고 싶다.

진단된 위험 (과거 볼트 분석에서 실증됨):
1. **순환 참조**: 자기 SNS 콘텐츠가 원출처(책·논문) 대신 출처로 기록된 메모 42건이 이미 존재.
2. **문체 오염**: 지식 문서에는 SNS 훅·이모지·팩폭 배제 원칙이 확정되어 있으나, 콘텐츠는 정확히 그 문체로 작성됨.
3. **에코 챔버**: 메모→콘텐츠→메모 무한 반복 시 시스템이 자기 출력물만 먹고 자라는 자기 복제.
4. **허위 인사이트**: AI가 "새 인사이트"라고 판단한 것이 실제로는 기존 메모의 재탕일 수 있음.

### 0-3. 도출된 해법: 역방향 4단 관문

콘텐츠 전체가 아니라 사람의 수정분(diff)만, 문체를 갈아입혀, 격리 상태(candidate)로 넣고, 사람이 아침에 승인해야만 정식 메모가 된다.

| 관문 | 기능 | 차단하는 오염 |
|------|------|--------------|
| 1. diff 추출 | AI 초안 vs 발행 최종본 비교, 사람이 추가·수정한 부분만 추출 (diff_learner.py 재사용) | 중복 지식 복제 |
| 2. 문체 변환 | SNS 문체 → 지식 서술문 변환 | 문체 오염 |
| 3. candidate 격리 | `status: candidate` 상태로 격리. 어떤 에이전트도 인용 불가 | 순환 참조·에코 챔버 |
| 4. 아침 승인 | 사람이 텔레그램에서 승인/수정/기각. 승인 시에만 정식 메모 승격 | 허위 인사이트 |

### 0-4. 텔레그램 단일 창구 결정

사용자 요구: "뭐든 텔레그램으로 확인받고 지시·소통." 드림그로우에 이미 확정된 텔레그램 봇(D-02)과 음성 승인 루프(Phase B)를 제텔카스텐 결재까지 확장한다. 봇을 2개 만들지 않고 1개 봇에 메시지 네임스페이스를 둔다.

### 0-5. 두 원본 문서 대조에서 발견한 충돌 → PART 3에서 해결

여기까지가 사고 과정이다. 이하는 결과물이다.

## PART 1. 텔레그램 통합 게이트웨이 설계

### 1-1. 원칙

1. **봇은 1개.** "드림그로우 비서 봇" 하나가 콘텐츠·지식·사례·시스템 알림을 모두 담당한다.
2. **텔레그램은 리모컨, 볼트가 본체.** 봇은 볼트 파일의 frontmatter만 바꾸고, 실제 동작(발행, 승격)은 기존 파이프라인이 상태 변화를 감지해 수행한다. 봇에 비즈니스 로직을 넣지 않는다.
3. **묶어서 보내고, 긴급만 즉시.** 아침 06:30 / 저녁 21:00 브리핑 2회 + 긴급(오류·발행실패)만 실시간. 알림 피로로 관문이 무너지는 것을 막는다.
4. 기존 안전 규칙(§3 명시적 확정 원칙, 회수 유예)은 지식 승인에도 동일 적용한다.

### 1-2. 메시지 네임스페이스

모든 메시지는 접두 태그와 콜백 네임스페이스를 갖는다. 이것이 두 시스템을 "한 창구에서 받되 절대 섞이지 않게" 하는 장치다.

| 태그 | 영역 | 콜백 접두 | 대상 볼트 |
|------|------|----------|----------|
| 📱[콘텐츠] | 초안 리뷰·발행 확정 | `ct:` | 드림그로우 |
| 🧠[지식] | candidate 메모 승인 | `zk:` | 초생산 |
| 📋[사례] | 사례은행 노랑 결재, Plaud 발화 승인 | `cs:` | 초생산 |
| ❓[질문] | review_queue, socrates 새벽 질문 | `rq:` | 초생산 |
| ⚙️[시스템] | 오류·발행실패·크론 이상 | `sys:` | 양쪽 |

### 1-3. 메시지 형식과 버튼 동작

#### A. 아침 브리핑 (06:30, 통합 1건 발송)

```
🌅 아침 브리핑 — 7월 5일 (토)

📱[콘텐츠] 리뷰 대기 2건
  1. "훈육이 무너지는 진짜 이유" (스레드, 오늘 19:00 예정)
     [✅ 확정] [✏️ 수정대기] [⏸ 보류]
  2. ...

🧠[지식] 승인 대기 1건
  어제 발행 콘텐츠에서 새 인사이트 후보:
  "훈육의 핵심 변수는 규칙의 내용이 아니라 적용의 예측가능성이다."
  원본 메모: P - 훈육은 처벌이 아니라 예측가능성이다
  [✅ 승인] [✏️ 수정 후 승인] [❌ 기각]

📋[사례] 노랑 결재 1건
  "선생님, 저는 틀리는 게 좋아요. 다시 할 수 있으니까." (3학년, 가명화 완료)
  [✅ 입고] [❌ 반려]

❓[질문] 오늘의 socrates 질문
  "규칙의 예측가능성이 중요하다면, 교실에서 예외를 허용해야
   할 순간은 언제입니까?"
  → 이 메시지에 답장(텍스트/음성)하면 직관+답변으로 기록됩니다.
     첫 줄은 직관으로 저장되며 에이전트는 개입하지 않습니다.
```

#### B. 저녁 브리핑 (21:00)

발행 결과 리포트(성공/실패, 채널별) + 내일 발행 초안 예고 + diff-learn 요약("오늘 학습된 수정 패턴 N건, 그중 지식 후보 M건은 내일 아침 결재 예정").

#### C. 즉시 알림 (조건 발생 시)

발행 실패, 크론 미실행, API 오류, Plaud 빨강 차단 통계("차단 2건 — 민감범주"; 내용은 절대 전송하지 않음).

#### D. 버튼 콜백 동작 정의

| 버튼 | 콜백 | 봇이 하는 일 |
|------|------|-------------|
| ✅ 확정 (콘텐츠) | `ct:confirm:<file_id>` | frontmatter `상태: 리뷰완료` + `발행시간` 기록. §3 회수 유예 유지 |
| ✏️ 수정대기 (콘텐츠) | `ct:revise:<file_id>` | `상태: 수정대기`. "수정 지시를 답장으로 보내주세요" 안내. 음성/텍스트 답장 → revision_loop 큐 + Honcho corrections 저장 |
| ✅ 승인 (지식) | `zk:approve:<memo_id>` | `status: candidate → approved`, `_system/candidates/` → 해당 단계 폴더로 이동, 원본 메모에 `used_in` 갱신 |
| ✏️ 수정 후 승인 (지식) | `zk:edit:<memo_id>` | "수정 내용을 답장으로" → 전사 → 메모 반영 → dry-run 미리보기 재푸시 → 재승인 시 승격 |
| ❌ 기각 (지식) | `zk:reject:<memo_id>` | 파일을 `_system/logs/rejected/`로 이동(삭제 아님), 기각 사유를 lessons.md에 누적 |
| ✅ 입고 (사례) | `cs:approve:<case_id>` | 사례은행 정식 입고, `author: 이한결(구술)`, `source_type: experience` |

#### E. 사람 → 시스템 방향 (지시·투입)

| 입력 | 라우팅 |
|------|--------|
| 브리핑 메시지에 **답장** | 답장 대상 message_id로 파일을 특정해 해당 파일에 반영 (아래 1-5 상태 DB) |
| `/사례 <내용>` 또는 음성+`사례` 키워드 | `6. 사례은행/_inbox/`에 저장 → 밤에 case-keeper 규격화 |
| `/메모 <내용>` | 초생산 1단계 인박스에 fleeting 메모 생성 (author: 이한결) |
| `/상태` | 오늘 크론 실행 현황, 대기 건수 요약 |
| 답장 아닌 단독 음성 | 봇이 "콘텐츠 수정 / 사례 / 메모 중 무엇입니까?" 버튼으로 되묻기 (오라우팅 방지) |

이 설계로 아이폰 옵시디언 없이도 침대·출근길·쉬는 시간에 두 시스템 전체를 운영할 수 있다.

### 1-4. 폴더 감시 규칙 (vault_watcher)

| # | 감시 대상 | 트리거 조건 | 전송 시점 | 쓰기 주체 |
|---|----------|------------|----------|----------|
| W1 | 드림그로우 `05 리뷰/` | `상태: 리뷰대기` 신규 | 아침 브리핑 (초안은 04:00 생성이므로) | team-content |
| W2 | 초생산 `_system/candidates/` | `status: candidate` 신규 | 아침 브리핑 | knowledge_feedback(신규, 21:30) |
| W3 | 초생산 `_system/review_queue.md` | 신규 항목 append | 아침 브리핑 | 전 에이전트 |
| W4 | 초생산 `_system/dialogues/<오늘>.md` | 파일 생성 | 아침 브리핑 | socrates (05:00) |
| W5 | 초생산 `6. 사례은행/_노랑대기/` | 신규 파일 | 아침 브리핑 | case-keeper |
| W6 | 발행 로그 (양쪽) | 실패 상태 | **즉시** | publisher류 |
| W7 | 크론 하트비트 | 예정 작업 미실행 30분 경과 | **즉시** | watchdog |

감시 방식: 5분 간격 폴링(파일 mtime + frontmatter 파싱). inotify는 Obsidian Sync의 부분 쓰기와 충돌 가능성이 있어 폴링을 기본으로 한다.

### 1-5. 상태 DB (봇의 유일한 자체 저장소)

`gateway_state.sqlite` 한 개. 테이블: `messages(message_id, file_path, namespace, sent_at, resolved)`. 용도는 "답장이 어느 파일에 대한 것인지" 매핑뿐이다. 콘텐츠·지식의 진실원천은 언제나 볼트 파일이며, DB가 날아가도 볼트는 무손상이다.

## PART 2. 두 시스템의 통합과 구분

### 2-1. 대원칙: "볼트는 둘, 창구는 하나, 다리는 좁게"

```
┌─ 초생산 볼트 (지식) ─────────┐          ┌─ 드림그로우 볼트 (콘텐츠) ──┐
│ 1~5단계 파이프라인            │  정방향   │ 01 아이디어 / 03 라이브러리  │
│ 6. 사례은행                  │ ───────▶ │ 05 리뷰 / 06 제작           │
│ _system/candidates/ ◀────────┼──────────┤ (zettel_reader가 읽기 전용)  │
│  (역방향 유일한 착륙장)        │  역방향   │ diff_learner → 4단 관문      │
└──────────────────────────────┘          └─────────────────────────────┘
                 ▲                                      ▲
                 └───────── 텔레그램 봇 1개 ─────────────┘
```

### 2-2. 통합하는 것 (공유 지점)

| 항목 | 통합 방식 |
|------|----------|
| 승인 창구 | 텔레그램 봇 1개, 네임스페이스로 구분 (PART 1) |
| 아침 루틴 | 제텔카스텐 인수인계 7장의 "아침 루틴 10분"을 06:30 텔레그램 브리핑에 흡수. 직관 한 줄 = 답장의 첫 줄 |
| 문체 학습 | Honcho(D-04)가 계속 심장. 지식 승인의 기각/수정 사유도 corrections에 저장해 knowledge_feedback의 문체 변환 품질을 학습 |
| 모델 정책 | D-08을 초생산 에이전트에도 적용: socrates·writer=fable 또는 opus, transplanter·classifier=sonnet, 단순 마킹 스크립트=haiku/파이썬 |
| Plaud | D-11 그대로: 크론(case-keeper)=CLI, 대화형=MCP |
| 서버·인프라 | 같은 우분투 서버, 같은 tmux, 같은 크론 체계. 크론 슬롯만 분리 (2-4) |
| 사례 공급 | **콘텐츠 필수 규칙의 "교실 에피소드"도 사례은행에서만 공급.** 드림그로우 writer도 사례 창작 금지, [사례 요청] 카드 방식 준용 |

### 2-3. 구분하는 것 (경계 규칙 — 완화 금지)

| # | 경계 규칙 |
|---|----------|
| B-1 | **쓰기 권한 매트릭스**: 드림그로우 에이전트가 초생산에 쓸 수 있는 곳은 `_system/candidates/` 단 하나. 초생산 에이전트는 드림그로우 볼트에 일절 쓰지 않는다. zettel_reader는 읽기 전용 |
| B-2 | **인용 자격**: zettel_reader가 콘텐츠 재료로 읽을 수 있는 메모 = 3~5단계 + `author: 이한결` + `_ai` 접두사 없음 + `status: candidate` 아님. (제텔카스텐 강철 규칙의 콘텐츠 확장) |
| B-3 | **문체 격리**: 초생산에는 SNS 훅·이모지·팩폭 문체 반입 금지(2관문에서 변환). 드림그로우 볼트의 콘텐츠 문체 규칙은 그대로 유지 |
| B-4 | **출처 사슬 관통**: candidate 출신 메모(`origin: content_feedback`)로 새 콘텐츠를 만들 때는 `source_memos`에 기록된 원래 메모와 그 뒤의 원출처(책·논문)까지 승계 인용. 자기 콘텐츠 URL을 출처로 쓰는 것 금지 → 순환 참조 42건 재발 방지 |
| B-5 | **기존 메모 자동 수정 금지**: 역방향 흐름은 기존 메모의 내용을 절대 덮어쓰지 않는다. 자동으로 할 수 있는 것은 `used_in: [발행URL, 날짜]` 이력 필드 추가뿐. 내용 병합은 승인 시 사람이 결정 |
| B-6 | **에이전트 소속 분리**: 드림그로우 4~5명(D-09) / 초생산 8명은 별도 팀. 겸직 금지. 유일한 신규 교량 요원 = knowledge_feedback(스크립트 수준, 에이전트 아님) |
| B-7 | **candidate는 어떤 에이전트도 인용 불가** — 양쪽 볼트 CLAUDE.md 공통 규칙에 명시 |

## PART 5. 병렬 검토 요청 포인트

검토 AI에게 요청하는 검증 항목:

1. **관문 우회 경로 존재 여부**: candidate 격리를 우회해 콘텐츠 파생 지식이 정식 메모가 되는 경로가 설계상 남아 있는가? (특히 revision_loop의 음성 수정이 candidate를 직접 승격시키는 시나리오)
2. **C-1 해결안의 타당성**: 초생산 모바일 입력을 텔레그램으로 대체하는 것이 사용자 워크플로우(수업 중 즉흥 사례 기록 등)를 실질적으로 저해하지 않는가?
3. **diff 추출의 한계**: 사람이 초안을 수정하지 않고 그대로 발행한 경우(diff=0) 인사이트 환류가 발생하지 않는데, 이때 놓치는 지식이 있는가? 반대로 단순 오탈자 수정이 인사이트로 오검출될 위험과 그 필터 기준은?
4. **텔레그램 단일 장애점(SPOF)**: 봇 다운 시 두 시스템의 승인이 모두 정지한다. 옵시디언 직접 결재를 예비 경로로 유지해야 하는가?
5. **크론 슬롯 21:30(knowledge_feedback)**: diff-learn(21:00) 완료를 보장하는 의존성 처리(파일 락 또는 완료 마커)가 필요한지.
6. **보안**: 텔레그램으로 전송되는 메모 요약에 학생 정보·민감 정보가 섞일 가능성과 그 필터(가명화 검사) 위치.
7. **누락 확인**: 두 원본 문서의 폐기 결정(D-01~D-14의 우측 열)을 부활시킨 항목이 없는지 전수 대조.
