### G-2. AGENTS.md — 규칙 하나만 구제하고 나머지는 아카이브

AGENTS.md의 콘텐츠 필수 규칙 3·4번이 CLAUDE.md보다 진화해 있습니다:

> (구) "'A가 아니라 B이다' 논증 구조" + "교실 경험 기반 에피소드 포함" — 매 콘텐츠 강제
> (신) "**주제 성격 → 적합한 구조 선택. 같은 구조 반복 금지**" + "'A가 아니라 B'와 교실 경험은 **주제에 맞을 때 쓰는 핵심 재료. 고정 순서/필수 템플릿 금지**"

같은 논증 틀을 모든 글에 강제하면 독자가 패턴을 읽어버리고, AI 생산 티가 납니다. 신규 규칙이 옳으므로 **CLAUDE.md 규칙 3·4를 이 문구로 교체**합니다 (봇 프로토타입의 SYSTEM_PROMPT에도 이미 신규 규칙이 들어가 있어, 이것이 더 나중의 판단임이 교차 확인됩니다).

Codex CLI 병행 체계(.codex/.agents/AGENTS.md 자체, bot의 /gpt 명령)는 **폐기(아카이브)**합니다. 이유: 도구 이원화는 규칙·스킬·훅을 두 벌로 유지하는 비용을 낳고, D-08 모델 정책(전 작업을 Claude 모델 4단으로 배치)과 충돌합니다. 단일 Claude Code 체제로 갑니다.

### G-3. telegram-claude-bot — Phase B가 "신규 개발"에서 "확장 개발"로 바뀝니다

이미 만들어진 것 (그대로 계승):

없어서 추가해야 하는 것 (v1 게이트웨이 설계 그대로):
- 인라인 버튼 콜백 (ct:/zk:/cs:/rq:/sys: 네임스페이스) 과 frontmatter 상태 변경
- 아침/저녁 브리핑, vault_watcher(W1~W7), 보이스 전사(faster-whisper), gateway_state.sqlite

**집행**: 백로그 B-2를 "telegram_gateway.py 신규 작성" → "**bot.py를 src/gateway/로 이동 후 확장**"으로 변경합니다. 개발량이 크게 줄어듭니다.

VPS 가이드에서 가져올 것: systemd 패턴(서버에서 봇은 systemd, 배치는 cron), 경로 환경변수화 지적(A-1과 동일 결론). 폐기할 것: VPS 신규 임대(D-01 옛 노트북 서버 유지, VPS는 노트북 실패 시 예비안), rclone/Drive로 코드 배포(git으로 단일화).

### G-4. 원소스 멀티유즈 기획서 — 파이프라인의 "입구"를 채우는 신규 Phase H

이 기획서는 지금까지 계획에 없던 **주제 발굴 리서치 엔진**을 정의합니다. 구조가 훌륭합니다:

- **1단 (주 1회, Manus Wide Research, 사람이 수동 실행)**: 교육 주제 1개를 7개 소스(유튜브·블로그·스레드·논문·아마존 도서·교육뉴스·맘카페)에서 병렬 조사 → 키워드 10~15개 + 근거 → **검수 관문 1: 선생님이 각도 선택**
- **2단 (매일, 기존 파이프라인)**: 선택 키워드 → 유튜브 롱폼 스크립트(=원본) → 릴스 3·스레드 9·뉴스레터 2로 파생 → **검수 관문 2: 발행 전 확인**

통합 판정 3가지:
1. **1단 채택**: 주 1회 수동 실행이므로 자동화 안전규칙과 충돌하지 않고, D-13(Manus 홈페이지 제작 폐기)과도 별개입니다. 산출물(키워드 리포트)은 볼트 `SNS 콘텐츠 제작 시스템/01 수집/`으로 들어와 collector 흐름에 합류합니다.
2. **"원본→파생" 원칙 채택**: "유튜브 롱폼이 원본이고 나머지는 파생"이라는 원칙을 team-content 규칙(rules/10)에 명문화합니다. 다섯 채널의 메시지가 한 줄기로 묶입니다.
3. **검수 관문 2개는 텔레그램으로 흡수**: 관문 1(키워드 선택)은 주 1회 브리핑에 버튼 선택형으로, 관문 2는 기존 ct: 결재와 동일합니다.

우선순위는 운영 안정화 후(3주차 이후)입니다. 지금의 병목은 리서치가 아니라 관제 계층 부재이기 때문입니다.

### G-5. .context 체크포인트 시스템 — 두 시스템 공통 계층으로 확대

CONTEXT(불변 배경)·STATE(진행률)·HANDOFF(다음 세션 인수인계)·CHANGELOG(변경 누적) 4파일 체계는 에이전트 OS의 "작업 기억"입니다. 이미 드림그로우 쪽에 도입되어 있으므로, **초생산 볼트 세션에도 동일 체계를 복제**합니다 (`초생산/_system/.context/`). Phase 1~2의 야간 배치(수백 개 파일 이식)는 여러 세션에 걸치므로, 이 체계가 없으면 매 세션 처음부터 상황 파악을 다시 해야 합니다.
