## 0. 목표와 대원칙

이한결(초등교사 22년, 교육성장네트워크 꿈들 대표, 새로운학교네트워크 운영진, 작가 지망, 경제적 자유 추구)의 두 시스템을 하나의 에이전트 OS로 통합합니다.

- **드림그로우 콘텐츠 자동화**: 지식 → 스레드·릴스·뉴스레터·카드뉴스·유튜브 → 자동 발행
- **제텔카스텐 세컨드 브레인**: 원료 → 5단계 파이프라인 → 책·논문

**플라이휠**: 지식이 콘텐츠를 낳고, 콘텐츠 제작 중 생긴 새 인사이트가 관문을 거쳐 지식으로 돌아옵니다. 콘텐츠를 만들수록 세컨드 브레인이 자랍니다.

**3대 대원칙**:
1. **볼트는 하나, 창구는 텔레그램 하나, 두 영역 사이 다리는 candidates 폴더 하나.**
2. **생각의 주인은 이한결입니다.** AI는 의견·주장·사례를 창작하지 않습니다. `_ai` 딱지 메모와 wiki는 4~5단계·집필에 인용 금지입니다.
3. **모든 파괴적 작업은 dry-run → 승인 → git 커밋 → 실행** 순서를 지킵니다.

### 1-4. 메타데이터 스키마 (인수인계 문서 3-2 계승)

`author`(메모 문장을 쓴 사람: 이한결|AI) ≠ `source_author`(지식의 원천). `_ai` 접두 메모는 4~5단계·집필 인용 금지, 이한결이 자기 언어로 고치면 승격 의식으로 `_ai` 제거. `verbatim: true`는 원문 그대로 이식. `source_type: own_content`는 "원출처 추적 필요" 자동 플래그(순환 참조 42건 재발 방지), `experience`는 책 사용 가능·논문에는 "[학술 근거 보강 필요]" 표시.

**Raindrop → clippings 이중 저장 설계 (D-31)**: 가능합니다. 그리고 이중 저장이 정답입니다. raindrop_sync(23:30 크론)가 신규 북마크를 발견하면 두 가지를 만듭니다.

1. **원문 전문 → `raw/clippings/`**: 웹 페이지 본문, 유튜브 자막 전문을 그대로 저장합니다 (frontmatter: 출처URL·수집일·Raindrop 태그). raw는 불변 원칙이므로 collector에게는 clippings/ 하위 **추가 전용** 권한만 부여합니다 — 기존 파일 수정·삭제는 여전히 금지입니다.
2. **활용 카드 → `SNS 시스템/01 수집/`**: sonnet이 요약·카테고리 판정·활용 각도 3줄을 붙인 작업용 카드를 만들고, 원문(raw/clippings 파일)을 [[링크]]로 겁니다.

이렇게 나누는 이유는 두 소비자의 요구가 다르기 때문입니다. 예를 들어 하버드 교육 대학원의 자기주도학습 기사를 Raindrop에 저장하면, 콘텐츠팀은 01 수집의 활용 카드("스레드 훅으로 쓸 만한 통계 2개, 훈육 카테고리")만 보면 되고, 훗날 책을 쓸 때 researcher는 raw/clippings의 원문 전문에서 정확한 인용문과 출처를 가져옵니다. 요약본만 남기면 책 쓸 때 원문이 없고, 원문만 남기면 콘텐츠팀이 매번 전문을 다시 읽어야 합니다.

## 3. 텔레그램 통합 게이트웨이 (승인·지시의 단일 창구)

**출발점**: 드라이브에서 발견된 telegram-claude-bot 프로토타입(bot.py)을 src/gateway/로 이식해 확장합니다. allowlist(TELEGRAM_ALLOWED_USER_IDS)·/run 명령 체계·systemd 상시 실행·긴 응답 분할은 계승하고, 아래를 추가합니다.

### 3-1. 원칙
1. 봇은 1개. 텔레그램은 리모컨, 볼트가 본체 — 봇은 frontmatter만 바꾸고 실제 동작은 파이프라인이 상태 변화를 감지해 수행합니다.
2. 묶어서 보내고 긴급만 즉시: 아침 06:30 + 저녁 21:00 브리핑, 오류·발행실패만 실시간. 알림 피로로 관문이 무너지는 것을 막습니다.
3. /run 수동 명령은 유지하되 **발행·승격은 반드시 버튼 결재를 경유**합니다 (명시적 확정 원칙·회수 유예 유지).

### 3-2. 메시지 네임스페이스

| 태그 | 영역 | 콜백 | 버튼 |
|------|------|------|------|
| 📱[콘텐츠] | 초안 리뷰·발행 확정 | `ct:` | [✅확정][✏️수정대기][⏸보류] |
| 🧠[지식] | candidate 메모 승인 | `zk:` | [✅승인][✏️수정 후 승인][❌기각] |
| 📋[사례] | 사례은행 노랑 결재·Plaud 발화 | `cs:` | [✅입고][❌반려] |
| ❓[질문] | review_queue·socrates 새벽 질문 | `rq:` | 답장(텍스트/음성) — 첫 줄은 직관(IL)으로 저장, 에이전트 개입 금지 |
| ⚙️[시스템] | 오류·발행실패·크론 이상·빨강 차단 통계(건수만) | `sys:` | — |

버튼 동작: ✅확정→`상태: 리뷰완료`+발행시간, ✏️수정대기→음성/텍스트 답장이 revision_loop 큐+Honcho corrections로, ✅승인→`candidate→approved`+해당 단계 폴더 이동+원본 메모 `used_in` 갱신, ❌기각→`_system/logs/rejected/` 이동(삭제 아님)+사유 lessons.md 누적, ✅입고→`author: 이한결(구술), source_type: experience`.

사람→시스템: 브리핑에 답장(message_id로 파일 특정), `/사례 <내용>`→사례은행 _inbox, `/메모 <내용>`→1. 메모 인박스, `/상태`→크론·대기 현황, 답장 아닌 단독 음성→"콘텐츠 수정/사례/메모 중 무엇입니까?" 되묻기.

### 3-3. 폴더 감시 규칙 (vault_watcher, 5분 폴링)

| # | 감시 대상 | 트리거 | 전송 |
|---|----------|--------|------|
| W1 | SNS 시스템 05 리뷰/ | `상태: 리뷰대기` 신규 | 아침 |
| W2 | _system/candidates/ | `status: candidate` 신규 | 아침 |
| W3 | _system/review_queue.md | 신규 항목 | 아침 |

| W6 | 발행 로그 | 실패 | **즉시** |

## 4. 역방향 4단 관문 (콘텐츠 → 지식 환류)

콘텐츠 전체가 아니라 사람의 수정분만, 문체를 갈아입혀, 격리 상태로, 사람의 승인을 거쳐 지식이 됩니다. 관문 하나가 오염원 하나씩을 전담합니다.

| 관문 | 동작 | 차단하는 오염 |
|------|------|--------------|
| 1. diff 추출 | AI 초안 vs 발행 최종본 비교, 사람이 추가·수정한 부분만 추출 (diff_learner.py 재사용) | 중복 지식 복제 |
| 2. 문체 변환 | SNS 훅·2인칭·감정 리듬 → 지식 서술문 재작성 (21:30 knowledge_feedback, fable) | 문체 오염 |
| 3. candidate 격리 | `status: candidate`로 candidates/ 격리. **어떤 에이전트도 인용·링크 불가** | 순환 참조(기존 42건)·에코 챔버 |
| 4. 아침 승인 | 텔레그램 zk: 버튼으로 승인/수정/기각. 승인 시에만 정식 메모 승격 | 허위 인사이트 |

candidate frontmatter: `status: candidate / origin: content_feedback / source_memos: [원본 메모] / published_url / author: human_edit_delta`

### 경계 규칙 B-1~B-7 (완화 금지)
- B-1 쓰기 교차점은 candidates/ 단 하나 (§1-3 매트릭스)
- B-2 콘텐츠 재료 자격: 3~5단계 + author: 이한결 + `_ai` 없음 + candidate 아님
- B-3 문체 격리: 제텔카스텐에 SNS 훅·이모지·팩폭 반입 금지
- B-4 출처 사슬 관통: content_feedback 메모로 새 콘텐츠 생성 시 source_memos의 원래 메모와 그 뒤 원출처(책·논문)까지 승계 인용. 자기 콘텐츠 URL 출처 금지
- B-5 기존 메모 자동 수정 금지: 자동 허용은 `used_in` 이력 추가뿐. 내용 병합은 승인 시 사람이 결정
- B-6 에이전트 소속 분리: 콘텐츠팀 4~5명 / 지식팀 8명, 겸직 금지. 유일한 교량 = knowledge_feedback(스크립트)
- B-7 candidate 인용 금지를 CLAUDE.md 공통 규칙에 명시
